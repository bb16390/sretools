# Kafka 适配器 - 产品需求文档

## 概述
- **摘要**: 在 Worker 端新增 Kafka 适配器，用于消费 Kafka 消息，并实现消费进度的实时上报，避免任务重启时重复消费。
- **目的**: 扩展数据收集能力，支持从 Kafka 消息队列中收集数据，并确保数据的可靠处理。
- **目标用户**: 使用 SRETools 的运维/开发人员，需要从 Kafka 收集数据的场景。

## 目标
1. 新增 Kafka 适配器，支持连接 Kafka 集群并消费指定主题的消息
2. 新增 Kafka 收集任务类型，可被任务调度器调度
3. 实现消费进度的实时上报和持久化
4. 支持任务重启时从上次提交的偏移量继续消费，避免重复消费
5. 支持消息转换和处理后上报到 Master 端

## 非目标 (Out of Scope)
- 不实现 Kafka 生产者功能
- 不支持 Kafka 集群的管理功能
- 不实现复杂的消息路由和处理规则引擎

## 背景与上下文
当前项目已有多种适配器（SQL、ClickHouse、InfluxDB、HTTP、Redis），用于从不同数据源收集数据。为了支持从消息队列中收集数据，需要新增 Kafka 适配器，使用 confluent-kafka 库实现。

## 功能需求
- **FR-1**: Kafka 适配器，继承自 AsyncBaseAdapter，支持配置 Kafka 集群地址、主题、消费者组等参数
- **FR-2**: Kafka 收集任务类型，继承自 BaseTask，支持被任务调度器调度
- **FR-3**: 消费进度实时上报和持久化，使用 Redis 或本地存储保存偏移量
- **FR-4**: 任务重启时从上次保存的偏移量继续消费
- **FR-5**: 支持消息转换（使用现有的 TransformExecutor）
- **FR-6**: 处理结果上报到 Master 端

## 非功能需求
- **NFR-1**: 支持至少 1000 条/秒的消息处理能力
- **NFR-2**: 消费进度上报延迟不超过 1 秒
- **NFR-3**: 任务重启时，能在 10 秒内恢复并继续消费
- **NFR-4**: 代码遵循项目现有的 Python 编码风格和架构模式

## 约束
- **技术**: 使用 confluent-kafka 库，Python 3.12+
- **业务**: 需保持与现有适配器和任务系统的架构一致性
- **依赖**: 需要安装 confluent-kafka 库，并可能需要 Redis 用于保存消费进度

## 假设
1. 用户已有可用的 Kafka 集群
2. Redis 适配器已可用，用于保存消费进度（可选）
3. 任务调度器和状态上报系统正常工作

## 验收标准

### AC-1: Kafka 适配器创建和连接
- **Given**: 用户提供 Kafka 集群配置
- **When**: 创建 Kafka 适配器实例
- **Then**: 适配器成功连接到 Kafka 集群
- **Verification**: programmatic
- **Notes**: 验证通过测试适配器是否能成功连接和订阅主题

### AC-2: Kafka 消息消费
- **Given**: Kafka 适配器已连接，任务已启动
- **When**: 主题中有新消息
- **Then**: 适配器能成功消费消息
- **Verification**: programmatic
- **Notes**: 验证消息被正确接收和处理

### AC-3: 消费进度上报和持久化
- **Given**: 任务正在消费消息
- **When**: 消费一批消息后
- **Then**: 消费进度（偏移量）被上报并持久化
- **Verification**: programmatic
- **Notes**: 验证偏移量被正确保存到存储中

### AC-4: 任务重启不重复消费
- **Given**: 任务已消费部分消息并保存了偏移量
- **When**: 任务重启后
- **Then**: 任务从上次保存的偏移量继续消费，不重复消费已处理的消息
- **Verification**: programmatic
- **Notes**: 验证重启后偏移量读取和消费恢复

### AC-5: 消息转换和结果上报
- **Given**: 配置了转换规则
- **When**: 消息被消费后
- **Then**: 消息被转换，结果上报到 Master 端
- **Verification**: programmatic
- **Notes**: 验证转换和上报流程

## 开放问题
1. 消费进度存储方式：使用 Redis 还是本地文件系统？（建议支持两种方式，默认使用 Redis）
2. 消息批处理大小和上报频率的配置参数需要确定
